# 文件传输后端数据模型设计

版本：v0.1  
状态：设计稿  
适用范围：小文件上传、凭码下载、管理员文件管理与预览

## 1. 设计目标

本系统是部署在服务器上的轻量文件传输服务，核心目标是低内存、低运维复杂度、可审计、可恢复。数据模型必须满足以下要求：

1. 所有用户都可以上传文件，上传成功后获得当前可用范围内唯一的文件码。
2. 非管理员不能查看文件列表，只能凭文件码下载对应文件。
3. 管理员可以查看当前已上传文件列表、查看文件码、下载、删除、批量删除，并预览 txt、md、pdf 等简单文件类型。
4. 文件内容存放在文件系统，元数据存放在 SQLite，避免将二进制文件塞入数据库。
5. 删除语义为删除服务器上的实际文件，同时保留元数据和事件明细；文件删除后，该文件码允许被后续上传重新使用。
6. 管理员可以登录并修改密码，初始密码为 `password123`。
7. 系统记录简单的上传、下载、删除事件明细，管理员可以查看并删除事件记录。

## 2. 关键设计决策

| 主题 | 决策 | 说明 |
| --- | --- | --- |
| 用户模型 | MVP 不建立普通用户表 | 非管理员是匿名访问者，不需要注册、登录、列表权限。 |
| 管理员模型 | 数据库存储单管理员密码哈希和会话 | 初始密码为 `password123`，首次启动初始化，后续可通过接口修改。 |
| 文件码 | 使用 6 位字母数字码，只要求当前可用文件唯一 | 文件码服务于便捷传文件；用户输入可大小写，服务端统一转大写；文件删除后允许复用。 |
| 文件存储 | 文件系统存储内容，SQLite 存储元数据 | 更符合低内存、低复杂度部署目标。 |
| 删除方式 | 文件物理删除，元数据软删除 | 管理端默认只展示可用文件；软删除记录用于历史查看和事件关联。 |
| 预览内容 | 不长期缓存预览正文 | txt/md 按需读取有限字节；pdf 由浏览器内联加载。 |
| 文件完整性 | 保存 SHA-256 | 支持完整性校验、ETag 和重复排查。 |
| 事件明细 | 独立 `file_events` 表记录 | 记录上传、下载、删除事件，管理员可删除事件记录。 |

## 3. 文件码规范

文件码是系统对外暴露的文件标识，也是非管理员下载文件的凭证。根据当前需求，文件码以易读、易输入为优先，不作为强安全凭证设计。

### 3.1 格式

推荐格式：

```text
^[0-9A-Za-z]{6}$
```

说明：

1. 文件码固定 6 位。
2. 用户输入字符集为数字 `0-9`、大写字母 `A-Z`、小写字母 `a-z`。
3. 服务端统一转换为大写后存储和查询，例如 `aB3dE9` 与 `AB3DE9` 视为同一个文件码。
4. 只要求当前 `available` 文件之间唯一。
5. 原文件删除后，该文件码允许被后续上传重新使用。

### 3.2 生成规则

1. 使用随机数从 36 个字符中生成 6 位文件码，字符集为 `0-9A-Z`。
2. 插入数据库时依赖“当前可用文件码唯一”的部分唯一索引兜底。
3. 如果新文件码与当前 `available` 文件冲突，重新生成并重试。
4. 已删除文件的历史记录可以保留相同文件码，不参与新上传的唯一性判断。

## 4. 实体模型

### 4.1 FileRecord

文件记录描述一份已上传文件的元数据。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | integer | 是 | SQLite 自增主键，仅内部使用。 |
| `code` | text | 是 | 对外文件码，6 位大写字母数字；输入可大小写；仅当前可用文件唯一，删除后可复用。 |
| `original_name` | text | 是 | 上传时的原始文件名，只作为展示和下载文件名。 |
| `stored_name` | text | 是 | 服务端存储文件名，不能由用户直接控制。 |
| `storage_path` | text | 是 | 相对数据目录的文件路径。 |
| `size_bytes` | integer | 是 | 文件大小，单位字节。 |
| `mime_type` | text | 是 | 服务端识别或兜底后的 MIME 类型。 |
| `extension` | text | 是 | 规范化后的小写扩展名，不含点。 |
| `sha256` | text | 是 | 文件内容 SHA-256 十六进制摘要。 |
| `preview_kind` | text | 是 | `none`、`text`、`markdown`、`pdf`。 |
| `status` | text | 是 | `available` 或 `deleted`。 |
| `uploaded_by_role` | text | 是 | `anonymous` 或 `admin`。 |
| `uploaded_at` | text | 是 | 上传时间，UTC RFC3339。 |
| `download_count` | integer | 是 | 成功下载次数。 |
| `last_downloaded_at` | text | 否 | 最近一次成功下载时间。 |
| `deleted_at` | text | 否 | 删除时间，UTC RFC3339。 |
| `deleted_by_role` | text | 否 | 删除者角色，MVP 只允许 `admin`。 |
| `created_at` | text | 是 | 元数据创建时间。 |
| `updated_at` | text | 是 | 元数据更新时间。 |

### 4.2 AdminUser

MVP 只支持一个管理员账号，用户名固定为 `admin`。首次启动时，如果数据库中不存在管理员记录，服务端创建初始管理员并将初始密码 `password123` 哈希后保存。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | integer | 是 | 管理员主键，MVP 固定为 `1`。 |
| `username` | text | 是 | 管理员用户名，MVP 固定为 `admin`。 |
| `password_hash` | text | 是 | 管理员密码哈希，禁止保存明文密码。 |
| `password_algo` | text | 是 | 密码哈希算法，例如 `argon2id` 或 `bcrypt`。 |
| `password_changed_at` | text | 是 | 最近一次修改密码时间。 |
| `created_at` | text | 是 | 创建时间。 |
| `updated_at` | text | 是 | 更新时间。 |

### 4.3 AdminSession

管理员登录成功后生成会话令牌。数据库只保存令牌哈希，接口返回明文令牌给客户端。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | integer | 是 | 会话主键。 |
| `admin_id` | integer | 是 | 对应管理员 ID。 |
| `token_hash` | text | 是 | 会话令牌哈希，唯一。 |
| `expires_at` | text | 是 | 过期时间。 |
| `revoked_at` | text | 否 | 主动退出或修改密码后的吊销时间。 |
| `last_used_at` | text | 否 | 最近一次使用时间。 |
| `created_at` | text | 是 | 创建时间。 |
| `updated_at` | text | 是 | 更新时间。 |

### 4.4 FileEvent

文件事件记录上传、下载、删除动作。事件记录面向管理员展示，可由管理员删除。删除事件记录不影响文件本身。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | integer | 是 | 事件主键。 |
| `file_id` | integer | 否 | 对应文件记录 ID；文件记录不存在时允许为空。 |
| `file_code` | text | 是 | 事件发生时的文件码快照。 |
| `original_name` | text | 否 | 事件发生时的原始文件名快照。 |
| `event_type` | text | 是 | `upload`、`download`、`delete`。 |
| `actor_role` | text | 是 | `anonymous`、`admin` 或 `system`。 |
| `admin_id` | integer | 否 | 管理员动作对应的管理员 ID。 |
| `result` | text | 是 | `success` 或 `failed`。 |
| `error_code` | text | 否 | 失败时的错误码。 |
| `ip_address` | text | 否 | 请求来源 IP。 |
| `user_agent` | text | 否 | 请求 User-Agent。 |
| `message` | text | 否 | 简短说明。 |
| `occurred_at` | text | 是 | 事件发生时间。 |
| `deleted_at` | text | 否 | 管理员删除该事件记录的时间。 |
| `deleted_by_admin_id` | integer | 否 | 删除事件记录的管理员 ID。 |

## 5. SQLite 表结构

### 5.1 files

```sql
CREATE TABLE files (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  code TEXT NOT NULL CHECK (
    length(code) = 6
    AND code GLOB '[0-9A-Z][0-9A-Z][0-9A-Z][0-9A-Z][0-9A-Z][0-9A-Z]'
  ),

  original_name TEXT NOT NULL CHECK (length(original_name) BETWEEN 1 AND 255),
  stored_name TEXT NOT NULL UNIQUE CHECK (length(stored_name) BETWEEN 1 AND 255),
  storage_path TEXT NOT NULL UNIQUE CHECK (length(storage_path) BETWEEN 1 AND 512),

  size_bytes INTEGER NOT NULL CHECK (size_bytes >= 0),
  mime_type TEXT NOT NULL DEFAULT 'application/octet-stream'
    CHECK (length(mime_type) BETWEEN 1 AND 127),
  extension TEXT NOT NULL DEFAULT ''
    CHECK (length(extension) <= 32),
  sha256 TEXT NOT NULL CHECK (length(sha256) = 64),

  preview_kind TEXT NOT NULL DEFAULT 'none'
    CHECK (preview_kind IN ('none', 'text', 'markdown', 'pdf')),
  status TEXT NOT NULL DEFAULT 'available'
    CHECK (status IN ('available', 'deleted')),

  uploaded_by_role TEXT NOT NULL DEFAULT 'anonymous'
    CHECK (uploaded_by_role IN ('anonymous', 'admin')),
  uploaded_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),

  download_count INTEGER NOT NULL DEFAULT 0 CHECK (download_count >= 0),
  last_downloaded_at TEXT,

  deleted_at TEXT,
  deleted_by_role TEXT CHECK (deleted_by_role IS NULL OR deleted_by_role = 'admin'),

  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),

  CHECK (
    (status = 'available' AND deleted_at IS NULL AND deleted_by_role IS NULL)
    OR
    (status = 'deleted' AND deleted_at IS NOT NULL)
  )
);
```

### 5.2 索引

```sql
CREATE INDEX idx_files_status_uploaded_at
  ON files (status, uploaded_at DESC, id DESC);

CREATE UNIQUE INDEX idx_files_available_code
  ON files (code)
  WHERE status = 'available';

CREATE INDEX idx_files_code
  ON files (code);

CREATE INDEX idx_files_original_name
  ON files (original_name);

CREATE INDEX idx_files_preview_kind
  ON files (preview_kind);

CREATE INDEX idx_files_sha256
  ON files (sha256);
```

索引说明：

1. `idx_files_status_uploaded_at` 支撑管理员默认列表页。
2. `idx_files_available_code` 保证当前可下载文件的文件码唯一，同时允许已删除记录保留相同文件码。
3. `idx_files_code` 支撑按文件码查询历史记录。
4. `idx_files_original_name` 支撑按文件名搜索。SQLite 默认不能高质量中文分词，MVP 仅支持 `LIKE` 模糊匹配。
5. `idx_files_preview_kind` 支撑管理端按可预览类型筛选。
6. `idx_files_sha256` 支撑后续重复文件检查或排查。

### 5.3 admin_users

```sql
CREATE TABLE admin_users (
  id INTEGER PRIMARY KEY CHECK (id = 1),
  username TEXT NOT NULL UNIQUE DEFAULT 'admin'
    CHECK (username = 'admin'),
  password_hash TEXT NOT NULL CHECK (length(password_hash) BETWEEN 20 AND 255),
  password_algo TEXT NOT NULL CHECK (password_algo IN ('argon2id', 'bcrypt')),
  password_changed_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
);
```

初始化规则：

1. 首次启动时，如果 `admin_users` 为空，创建 `id=1`、`username='admin'` 的记录。
2. 初始密码为 `password123`，必须先哈希再写入数据库。
3. 已存在管理员记录时，启动过程不得覆盖已有密码。

### 5.4 admin_sessions

```sql
CREATE TABLE admin_sessions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  admin_id INTEGER NOT NULL,
  token_hash TEXT NOT NULL UNIQUE CHECK (length(token_hash) = 64),
  expires_at TEXT NOT NULL,
  revoked_at TEXT,
  last_used_at TEXT,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
  updated_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),

  FOREIGN KEY (admin_id) REFERENCES admin_users(id) ON DELETE CASCADE
);
```

```sql
CREATE INDEX idx_admin_sessions_token_hash
  ON admin_sessions (token_hash);

CREATE INDEX idx_admin_sessions_admin_expires
  ON admin_sessions (admin_id, expires_at DESC);
```

### 5.5 file_events

```sql
CREATE TABLE file_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  file_id INTEGER,
  file_code TEXT NOT NULL CHECK (
    length(file_code) = 6
    AND file_code GLOB '[0-9A-Z][0-9A-Z][0-9A-Z][0-9A-Z][0-9A-Z][0-9A-Z]'
  ),
  original_name TEXT CHECK (original_name IS NULL OR length(original_name) BETWEEN 1 AND 255),

  event_type TEXT NOT NULL
    CHECK (event_type IN ('upload', 'download', 'delete')),
  actor_role TEXT NOT NULL
    CHECK (actor_role IN ('anonymous', 'admin', 'system')),
  admin_id INTEGER,

  result TEXT NOT NULL DEFAULT 'success'
    CHECK (result IN ('success', 'failed')),
  error_code TEXT,
  ip_address TEXT,
  user_agent TEXT,
  message TEXT,

  occurred_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
  deleted_at TEXT,
  deleted_by_admin_id INTEGER,

  FOREIGN KEY (file_id) REFERENCES files(id) ON DELETE SET NULL,
  FOREIGN KEY (admin_id) REFERENCES admin_users(id) ON DELETE SET NULL,
  FOREIGN KEY (deleted_by_admin_id) REFERENCES admin_users(id) ON DELETE SET NULL
);
```

```sql
CREATE INDEX idx_file_events_visible_occurred
  ON file_events (deleted_at, occurred_at DESC, id DESC);

CREATE INDEX idx_file_events_type_occurred
  ON file_events (event_type, occurred_at DESC, id DESC);

CREATE INDEX idx_file_events_file_id
  ON file_events (file_id);

CREATE INDEX idx_file_events_file_code
  ON file_events (file_code);
```

事件索引说明：

1. `idx_file_events_visible_occurred` 支撑管理员默认事件列表。
2. `idx_file_events_type_occurred` 支撑按上传、下载、删除类型筛选。
3. `idx_file_events_file_id` 和 `idx_file_events_file_code` 支撑按文件查事件。

## 6. SQLite 运行参数

服务启动时应设置以下 PRAGMA：

```sql
PRAGMA foreign_keys = ON;
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA busy_timeout = 5000;
```

说明：

1. `WAL` 提升读写并发，适合管理端列表与上传下载并存的场景。
2. `synchronous=NORMAL` 在可靠性和性能之间取平衡。
3. `busy_timeout` 避免短时间写锁竞争直接失败。

## 7. 文件系统布局

推荐数据目录：

```text
DATA_DIR/
  app.db
  uploads/
    2026/
      05/
        08/
          AB3DE9-K7P2MQ8Z.txt
  tmp/
  trash/
```

规则：

1. `DATA_DIR` 由环境变量配置，默认可为 `/data`。
2. 上传先写入 `tmp/`，校验和元数据准备完成后原子移动到 `uploads/`。
3. `storage_path` 只保存相对路径，例如 `uploads/2026/05/08/AB3DE9-K7P2MQ8Z.txt`。
4. 用户上传的文件名绝不参与实际路径拼接，只用于 `original_name` 和下载响应头。
5. 删除时先将文件原子移动到 `trash/`，数据库提交成功后再清理 `trash/` 中文件。
6. 因为文件码允许复用，`stored_name` 不应只使用文件码，必须增加随机后缀或内部 ID，避免历史记录路径冲突。

## 8. 预览类型判定

`preview_kind` 由服务端根据扩展名和 MIME 类型共同判定。

| `preview_kind` | 条件 | 预览方式 |
| --- | --- | --- |
| `text` | `.txt` 或 `text/plain` | 管理端按需读取有限字节并返回文本。 |
| `markdown` | `.md`、`.markdown` 或 `text/markdown` | 管理端按需读取有限字节并返回 Markdown 原文。 |
| `pdf` | `.pdf` 或 `application/pdf` | 管理端以 inline PDF 流方式返回。 |
| `none` | 其他类型 | 不提供预览，只允许下载。 |

文本预览建议限制：

1. 默认最多读取前 `1 MiB`，超出时返回 `truncated=true`。
2. 优先按 UTF-8 解码，允许 UTF-8 BOM。
3. 对 `.txt` 和 `.md` 可尝试 GB18030 回退，以便兼容中文文本文件。
4. 预览接口只面向管理员。

## 9. 状态与约束

### 9.1 文件状态机

```text
uploading -> available -> deleted
```

状态说明：

1. `uploading` 是过程状态，不落库。上传完成前不能生成可下载记录。
2. `available` 表示文件存在于 `storage_path`，可以凭码下载。
3. `deleted` 表示服务器上的实际文件已删除。管理员默认列表不展示，凭码下载返回不存在。

### 9.2 不变量

1. `files.code` 在 `status='available'` 的文件中唯一；已删除文件的 code 不参与唯一性约束。
2. `status='available'` 时，`deleted_at` 和 `deleted_by_role` 必须为空。
3. `status='deleted'` 时，`deleted_at` 必须非空。
4. `available` 文件必须能在 `DATA_DIR/storage_path` 找到。
5. 服务端不得信任客户端提交的 MIME 类型、扩展名、文件名和路径。
6. 非管理员没有任何查询文件集合的能力。
7. `file_events` 记录删除后只从管理员默认列表中隐藏，不影响对应文件记录。

## 10. 事务设计

### 10.1 上传事务

推荐流程：

1. 接收 multipart 文件流，写入 `tmp/` 临时文件，同时计算 `sha256` 和 `size_bytes`。
2. 校验文件大小、文件名长度、MIME 类型和扩展名。
3. 生成 6 位文件码和最终 `storage_path`。
4. 开启 SQLite 写事务。
5. 将临时文件原子移动到最终路径。
6. 插入 `files` 记录，状态为 `available`。
7. 插入 `file_events` 上传成功记录。
8. 提交事务。
9. 如果文件码与当前可用文件冲突，重新生成文件码并重试。
10. 任意步骤失败时，清理临时文件或最终文件。

### 10.2 删除事务

推荐流程：

1. 开启 SQLite 写事务。
2. 管理员删除时按 `file_id` 查询文件记录并加写锁语义控制。
3. 如果记录不存在，返回 `not_found`。
4. 如果记录已经是 `deleted`，返回幂等成功。
5. 将实际文件从 `uploads/` 原子移动到 `trash/`。
6. 更新 `status='deleted'`、`deleted_at`、`deleted_by_role='admin'`、`updated_at`。
7. 插入 `file_events` 删除成功记录。
8. 提交事务。
9. 提交后删除 `trash/` 中的文件。
10. 如果提交后物理删除失败，保留在 `trash/`，由启动清理或定时清理处理。
11. 删除完成后，该文件码可以被新上传文件重新使用。

### 10.3 下载计数

成功返回文件流后更新：

```sql
UPDATE files
SET download_count = download_count + 1,
    last_downloaded_at = strftime('%Y-%m-%dT%H:%M:%fZ', 'now'),
    updated_at = strftime('%Y-%m-%dT%H:%M:%fZ', 'now')
WHERE code = ? AND status = 'available';
```

如果需要严格统计完整下载次数，应在响应写入完成后更新；如果实现复杂，MVP 可以统计成功打开文件并开始响应的次数。

下载成功时同时插入一条 `file_events` 下载成功记录。管理员下载也记录为 `download`，`actor_role='admin'`；非管理员凭码下载记录为 `actor_role='anonymous'`。

## 11. 事件记录设计

### 11.1 记录范围

MVP 记录以下事件：

| 事件类型 | 触发时机 | 关键字段 |
| --- | --- | --- |
| `upload` | 文件上传成功后 | `file_id`、`file_code`、`original_name`、`actor_role`、`ip_address`、`user_agent`。 |
| `download` | 文件下载成功打开并开始响应后 | `file_id`、`file_code`、`original_name`、`actor_role`、`admin_id`。 |
| `delete` | 管理员删除文件成功后 | `file_id`、`file_code`、`original_name`、`admin_id`。 |

失败事件可以按实现成本选择性记录。若记录失败事件，`result='failed'`，并填写 `error_code`。

### 11.2 删除事件记录

管理员删除事件记录时，推荐执行软删除：

```sql
UPDATE file_events
SET deleted_at = strftime('%Y-%m-%dT%H:%M:%fZ', 'now'),
    deleted_by_admin_id = ?
WHERE id = ? AND deleted_at IS NULL;
```

说明：

1. 事件记录删除后，管理员默认事件列表不再展示。
2. 删除事件记录不删除文件，也不改变 `files.download_count` 等统计字段。
3. 事件记录删除接口应保证幂等，重复删除同一条记录仍视为成功。

## 12. 清理与恢复

服务启动时建议执行轻量清理：

1. 删除 `tmp/` 中超过 24 小时的临时文件。
2. 删除 `trash/` 中超过 24 小时的文件。
3. 扫描 `available` 记录对应文件是否缺失，记录错误日志但不自动改状态。
4. 扫描磁盘孤儿文件，记录错误日志但不自动删除，避免误删。
5. 删除或吊销过期的 `admin_sessions`。

## 13. 配置项

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| `DATA_DIR` | `/data` | SQLite 和上传文件的根目录。 |

| `MAX_UPLOAD_BYTES` | `52428800` | 单文件最大上传大小，默认 50 MiB。 |

| `PUBLIC_BASE_URL` | 空 | 可选，用于生成完整下载 URL。 |

管理员初始账号：

| 项 | 值 |
| --- | --- |
| 用户名 | `admin` |
| 初始密码 | `password123` |

说明：初始密码仅用于首次初始化。数据库已经存在管理员记录时，启动过程不得重置密码。

## 14. 后续扩展预留

当前模型优先服务 MVP。后续可以在不破坏现有 API 的基础上扩展：

1. `expires_at`：文件自动过期。
2. `password_hash`：单文件下载密码。
3. 多管理员账号和权限分级。
4. 更详细的审计日志。
5. `tags`：管理员侧标签分类。
6. 事件记录导出和自动清理策略。
